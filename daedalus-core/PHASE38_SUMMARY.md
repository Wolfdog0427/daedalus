# Phase 38: Tier-3 Policy Engine

## What Was Added

- **Policy registry**: an in-memory registry of condition/action pairs that
  can automatically generate Tier-3 proposals or execution plans when their
  conditions are met — but never execute them.
- **Condition model**: three read-only, side-effect-free condition types
  that evaluate against system state:
  - `setting_equals` — matches a runtime setting value
  - `drift_above` — compares an autonomy drift metric against a threshold
  - `proposal_count` — checks how many proposals exist in a given status
- **Policy actions**: two creation-only action types:
  - `create_typed_proposal` — creates a governed proposal with
    `policy_origin` evidence for auditability
  - `create_execution_plan` — creates a plan referencing existing proposals
- **Evaluation engine**: an operator-triggered `evaluate_policies` function
  that iterates enabled policies, evaluates conditions, performs actions on
  match, and logs all results.
- **Cockpit commands**: `t3_policy_create`, `t3_policies`, `t3_policy`,
  `t3_policy_enable`, `t3_policy_disable`, `t3_policy_eval`.
- **Dashboard section**: "Tier-3 Policies" showing enabled/disabled state,
  last evaluation result, and proposal/plan generation counts.

## Why It Is Safe

- Policies create proposals and plans through the existing governed helpers —
  no new mutation pathways.
- Policies never execute proposals or plans; all execution remains
  operator-triggered through existing approval gates.
- Condition evaluators are pure, read-only functions that never modify state.
- Unknown condition or action types are rejected at creation time.
- Evaluation only runs when the operator explicitly invokes it; there is no
  background or scheduled execution.
- All policy-generated artifacts carry `policy_origin` metadata for
  traceability.

## How It Prepares for the Next Phase

- The evaluation engine is a natural hook for scheduled evaluation cadences
  or policy windows (time-bounded activation).
- Condition types can be extended with telemetry-derived signals (error rate,
  latency thresholds) for adaptive tuning.
- The `policy_origin` field enables audit trails and rollback scoping —
  "undo everything generated by policy X."
- The registry supports enable/disable toggling, which is the foundation for
  policy lifecycle management and operator overrides.
